# 🛡 Правила изменения кода

## 1. Не навреди (Chesterton's Fence)
Меняй ТОЛЬКО то, что просили. Не удаляй, не рефакторь, не «оптимизируй» код, не относящийся к задаче.
Рефакторинг = отдельная задача с подтверждением пользователя.

## 2. Scope Lock (Scope Creep Prevention)
Перед работой обозначь список затрагиваемых файлов.
Изменения за пределами — только с подтверждения пользователя.

## 3. Лимиты (Clean Code, Google Style Guide)
- Функция ≤ 80 строк
- Файл ≤ 300 строк
- Класс ≤ 200 строк
- Превышение → декомпозиция на приватные `_helper`-функции или пакет с `__init__.py`

## 4. Слои (SOLID SRP, Two Scoops of Django)
- **Views** = HTTP-обёртка
- **Services** = бизнес-логика
- **Models** = данные
- Если view содержит `for`, математику или ORM-запросы кроме `.get()`/`.filter()` — вынести в `services/`

## 5. Уточнения
При неоднозначной задаче — задать уточняющие вопросы **ПЕРЕД** началом работы, а не угадывать намерение.

# 🔍 Обязательный аудит и тестирование

**Применяется ВСЕГДА, к любой работе:**

1. **Перед изменением** — прочитать и понять существующий код
2. **После изменения** — запустить тесты / проверить работоспособность
3. **Логирование** — убедиться, что изменения видны в логах / отладке
4. **Регрессия** — проверить, что существующий функционал не сломался
5. **Документирование** — зафиксировать что изменено и почему
